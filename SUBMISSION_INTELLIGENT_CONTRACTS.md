## What this contract is

A prediction market primitive built entirely in Python on GenLayer. It covers the full lifecycle: market creation with AI-set opening odds, point-based betting pools, AI-adjudicated resolution, cancellation with refunds, and an embedded three-game points casino. Everything runs through GenLayer's consensus layer with no external oracle and no off-chain resolution step.

The contract is self-upgrading (`upgrade(new_code: bytes)`), meaning the entire codebase can be hot-swapped without migrating storage or redeploying dependent state.

---

## Why the consensus design is non-trivial

This contract uses two different GenLayer equivalence principles on purpose, and the choice matters.

**`gl.eq_principle.strict_eq`** is used for market resolution and game randomness. When resolving a market, every validator independently fetches the evidence URL and asks the same question: what is the outcome? The answers must match exactly. This is the right primitive for binary factual questions where "close enough" is not acceptable. If validators disagree on who won, the transaction does not finalize.

**`gl.eq_principle.prompt_non_comparative`** is used for generating market questions and setting opening odds. A daily crypto market question generated by two validators will never be word-for-word identical, and the exact wording does not matter as long as it meets the criteria (valid JSON, resolvable question, probabilities sum to 100). This is the right primitive when you need AI-generated content to pass a quality bar rather than an identity check.

Using `strict_eq` for both would break market creation (AI text is non-deterministic). Using `prompt_non_comparative` for resolution would be wrong (validators could accept different winners). The contract draws that distinction explicitly and consistently.

A third consensus call handles randomness for the games:

```python
def get_entropy() -> str:
    resp = gl.nondet.web.get("https://api.drand.sh/public/latest")
    data = json.loads(resp.body.decode("utf-8", "replace"))
    return str(data.get("randomness", ""))

beacon = gl.eq_principle.strict_eq(get_entropy)
```

All validators independently fetch the drand beacon and must agree on the same round value before any game result is committed. The beacon is mixed into a deterministic hash along with the player address and a global nonce. This is documented in the source as harm-reduction (drand is public, not a cryptographic guarantee) but it is a real step up from pure nonce-based outcomes and shows the pattern for integrating external randomness through `strict_eq`.

## State design

```python
owner:          str
market_count:   u64
game_nonce:     u64
markets:        TreeMap[str, str]   # market_id -> JSON blob
market_ids:     DynArray[str]
last_daily:     u64
last_weekly:    u64
last_monthly:   u64
lb_stats:       TreeMap[str, str]   # address -> JSON stats
lb_addresses:   DynArray[str]
points:         TreeMap[str, str]   # address -> JSON {balance, claimed}
claimed:        TreeMap[str, str]
last_game:      TreeMap[str, str]   # address -> last game result JSON
usernames:      TreeMap[str, str]   # address -> username
username_index: TreeMap[str, str]   # username.lower() -> address
```

Markets store all bet state inline as nested JSON inside a single TreeMap entry. This avoids per-bet storage slots and makes the full market readable in one `gen_call`. The tradeoff is that markets with many bettors produce larger blobs, which is fine at testnet scale.

The leaderboard is maintained incrementally: `_add_xp` is called at the end of every bet resolution and every game outcome (wins and losses), so the leaderboard reflects a running total without needing a scan at read time.

## AI methods and how they compose

```
create_market / create_daily_market / create_weekly_market / create_monthly_market
    calls _generate_odds (prompt_non_comparative)
    OR _ai_generate_market (prompt_non_comparative)  <-- single call: question + odds together

resolve_market
    calls get_verdict (strict_eq)
    reads: evidence_url, question, outcomes from market state
    returns: "YES", "NO", or one of the outcome strings, or "PENDING"

play_coinflip / play_dice / play_rps
    calls get_entropy (strict_eq) inside _roll
    mixes: address + salt + game_nonce + drand_beacon
```

One constraint worth documenting for other builders: functions passed to `prompt_non_comparative` or `strict_eq` must be minimal closures capturing a single pre-built value. GenVM serializes these closures to distribute to validators and crashes with `exit_code 1` on multi-variable captures. The working pattern is:

```python
full_prompt = build_the_entire_prompt_here(...)

def gen() -> str:
    return str(gl.nondet.exec_prompt(full_prompt)).strip()

raw = gl.eq_principle.prompt_non_comparative(gen, task=..., criteria=...)
```

Not this:

```python
def gen() -> str:
    # capturing topic_prompt, deadline_str, market_count from outer scope
    prompt = topic_prompt + "..." + deadline_str + "..." + str(market_count)
    return str(gl.nondet.exec_prompt(prompt)).strip()
```

The second pattern causes `exit_code 1`. The source code documents this in comments so other builders know why.

## Points system

Points are stored as a JSON blob per address rather than a raw integer:

```json
{"balance": 1500, "claimed": true}
```

This lets the contract check whether a wallet has already claimed starter points without a separate mapping. Transfer, claim, and game payout all go through `_update_points(addr, delta)` which handles the parse/update/serialize cycle atomically within a single contract call.

## What other builders can reuse

The contract demonstrates a complete set of patterns that are not covered by simple GenLayer examples:

1. How to choose between `strict_eq` and `prompt_non_comparative` for different AI tasks within the same contract
2. How to integrate external data (`gl.nondet.web.get`) inside a `strict_eq` call for consensus-verified randomness
3. How to build minimal closures that survive GenVM serialization
4. How to maintain a live leaderboard as a side effect of write operations rather than computing it at read time
5. How to handle the `_now()` fallback pattern when `gl.message_raw["timestamp"]` is unavailable
6. How to structure multi-step financial operations (bet, resolve, claim, refund) with proper state guards to prevent double-claims

The contract source is fully commented at each of these points.

---

## Known limitations (documented in source)

`_now()` falls back to `0` when `gl.message_raw["timestamp"]` is not populated. All time-dependent logic uses `now > 0` guards so the contract works correctly even when the timestamp is unavailable.

Game randomness is drand-mixed but not commit-reveal. A player who fetches the current drand round before submitting could in principle precompute outcomes. The source documents this explicitly and notes that a two-transaction commit-reveal would be required for games handling real value.
